# 高併發與資料安全分析報告

## 1. 潛在的資料安全性與高併發風險
在檢視電子書城專案的 `bookstore.service.OrderService` 結帳與建立訂單邏輯時，發現在處理購物車轉訂單 (`createOrderFromCart`) 以及後台新增訂單 (`insertOrder`) 時，存在嚴重的**超賣 (Overselling) 與遺失更新 (Lost Update)** 風險。

目前扣除庫存的作法是直接從 `Cart` 關聯中取出 `BooksBean`，或是透過無鎖的 `bookRepository.findById` 取出實體，檢查庫存在應用程式記憶體中是否足夠後，再寫回資料庫。
這種作法在單執行緒下沒有問題，但在**高併發環境（例如促銷活動、限量書籍搶購）**下，會違反 ACID 屬性中的隔離性 (Isolation)。

### 嚴重的風險場景
假設某本暢銷書庫存僅剩 1 本，此時使用者 A 與使用者 B 同時進行結帳：
1. **讀取階段：** 執行緒 A 與執行緒 B 同時從資料庫讀取該書庫存，皆為 `1`。
2. **檢查階段：** 兩者皆判斷 `1 >= 1`，允許結帳。
3. **寫入階段：** 執行緒 A 扣減庫存為 0 並存入資料庫；接著執行緒 B 亦將庫存扣減為 0 並覆蓋資料庫。
4. **結果：** 產生了兩筆訂單，但庫存卻只扣了一次（或是變成負數），造成**超賣現象**，商店無法如期交貨，導致客訴與財務損失。若扣款邏輯也未上鎖，甚至可能發生物品售出卻未確實扣款的嚴重異常。

---

## 2. 鎖的使用機制是否需調整？
**需要。**
為了解決高併發下的競爭條件 (Race Condition)，必須引入資料庫層級的鎖機制。針對庫存扣減這類敏感寫入操作，最常見的解法是使用**悲觀寫鎖 (Pessimistic Write Lock)**。

所幸專案內的 `BookRepository` 已定義了帶有悲觀寫鎖的查詢方法：
```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT b FROM BooksBean b WHERE b.bookId = :id")
Optional<BooksBean> findByIdWithLock(@Param("id") Integer id);
```
但 `OrderService` 在結帳時卻**未使用它**。我們只需要調整 `OrderService`，在扣除庫存及驗證前，一律透過 `findByIdWithLock` 鎖定該筆書籍記錄，確保同一時間只有一個交易(Transaction)能讀取並修改該本書的庫存。

## 3. 修改前後的 Code 差異

### ❌ 修改前 (有超賣風險的做法)
在 `OrderService.java` 的 `createOrderFromCart` 方法中：

```java
// 3. 檢查庫存 & 計算總金額
for (Cart cart : cartItems) {
    // 【問題點】直接從 Cart 記憶體中取得 BooksBean，未對資料庫紀錄上鎖
    BooksBean book = cart.getBooksBean();

    // 檢查庫存
    if (book.getStock() < cart.getQuantity()) {
        throw new BusinessException(400, "庫存不足");
    }

    // 扣除庫存
    book.setStock(book.getStock() - cart.getQuantity());
    bookRepository.save(book); // 發生併發時可能覆寫別人的扣減結果
}
```

### ✅ 修改後 (使用悲觀寫鎖保護資料正確性)
調整 `OrderService.java`，強制透過 `findByIdWithLock` 取出並鎖定資源：

```java
// 3. 檢查庫存 & 計算總金額
for (Cart cart : cartItems) {
    // 【修正】使用帶有 PESSIMISTIC_WRITE 的查詢，鎖定該書籍直到 Transaction 結束
    BooksBean book = bookRepository.findByIdWithLock(cart.getBooksBean().getBookId())
            .orElseThrow(() -> new BusinessException(404, "找不到書籍"));

    // 檢查庫存（此時若有其他執行緒嘗試讀取該書，會被阻擋等待解鎖）
    if (book.getStock() < cart.getQuantity()) {
        throw new BusinessException(400, "庫存不足");
    }

    // 扣除庫存
    book.setStock(book.getStock() - cart.getQuantity());
    bookRepository.save(book); 
}
```

同樣的鎖定機制也應套用至 `UserCouponBean` （優惠券）的驗證與更新上，防止使用者利用併發漏洞重複折抵同一張優惠券。
