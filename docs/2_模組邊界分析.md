# 模組邊界分析報告

## 1. 模組邊界是否過於模糊與逾越？
經過檢視專案架構，我們發現系統中的**模組職責邊界存在嚴重過於模糊與逾越**的狀況。這主要表現在服務層 (Service Layer) 之間缺乏封裝，**Service 直接依賴並操作其他領域的 Repository (資料層)**。

以 `OrderService` 為例，該服務不僅僅負責「訂單」業務本身，更直接注入並操作以下儲存庫：
- `CartRepository` (購物車模組)
- `BookRepository` (書籍/商品模組)
- `UserRepository` (使用者模組)
- `UserCouponRepository` (折價券模組)

### 會產生什麼嚴重的問題？
1. **高耦合度 (High Coupling)**：訂單模組與其他領域的資料表結構緊密綁定。若購物車 (`Cart`) 的表格結構需要優化，或是書籍庫存改由 Redis 獨立管理，除了修改 `Cart` / `Book` 模組外，連 `OrderService` 都會受到波及並需大量改寫。
2. **貧血領域模型與邏輯分散**：驗證優惠券、扣款、清空購物車的核心業務邏輯不在專屬的模組中處理，全被硬塞在 `OrderService`，導致程式碼臃腫且無法重用。
3. **微服務化困難**：未來如果網站流量變大，想要將系統拆分為「訂單中心」、「會員中心」與「商品中心」等獨立服務時，目前的架構將是致命傷，因為跨領域的資料庫操作會讓拆分極度困難。

---

## 2. 該如何解決？
套用**領域驅動設計 (DDD)** 或是**服務導向架構 (SOA)** 的思維，嚴格遵守**單一職責原則 (SRP)**。
- 每個 Service 只能注入「自己管理」的 Repository。
- 當需要其他模組的資料或操作時，**必須呼叫該模組的 Service 所提供的公開方法 (API/Interface)**。
- 由上層服務（或 Facade 門面服務）負責調度各個 Service 來完成一個跨邊界的作業。

例如：`OrderService` 不該直接呼叫 `cartRepository.deleteByUserId(userId)`，而應該呼叫 `cartService.clearCart(userId)`。

---

## 3. 修改前後的 Code 差異

### ❌ 修改前 (高耦合、逾越邊界的做法)
在 `OrderService.java` 中直接注入其他模組的 Repository：

```java
@Service
@Transactional
public class OrderService {
    @Autowired private OrdersRepository ordersRepository;
    @Autowired private OrderItemRepository orderItemRepository;
    
    // 【問題點】逾越邊界，直接依賴其他領域的 Repository
    @Autowired private CartRepository cartRepository;
    @Autowired private BookRepository bookRepository;
    @Autowired private UserCouponRepository userCouponRepository;

    public Orders createOrderFromCart(Integer userId, CheckoutRequest request) {
        // ... 直接跨領域存取與操作資料
        List<Cart> cartItems = cartRepository.findByUserId(userId);
        
        // ... 直接在訂單領域扣會員的優惠券
        UserCouponBean userCoupon = userCouponRepository.findById(request.getCouponId()).get();
        userCoupon.setStatus(1);
        userCouponRepository.save(userCoupon);

        // ... 直接去刪除購物車
        cartRepository.deleteByUserId(userId);
    }
}
```

### ✅ 修改後 (明確邊界，透過 Service 互相溝通)
重構依賴關係，由各自的 Service 管理其領域邏輯：

```java
@Service
@Transactional
public class OrderService {
    @Autowired private OrdersRepository ordersRepository;
    @Autowired private OrderItemRepository orderItemRepository;
    
    // 【修正】只注入其他領域的 Service，不接觸其 Data Access Layer (Repository)
    @Autowired private CartService cartService;
    @Autowired private BookService bookService;
    @Autowired private UserCouponService userCouponService;

    public Orders createOrderFromCart(Integer userId, CheckoutRequest request) {
        // 1. 透過 CartService 獲取購物車內容 (CartService 內部去查 Repository)
        List<CartDTO> cartItems = cartService.getUserCartItems(userId);
        
        // 2. 透過 BookService 扣減庫存 (BookService 內部封裝了悲觀鎖與庫存檢查)
        bookService.deductStockForCheckout(cartItems);

        // 3. 建立訂單邏輯 (OrderService 專注於自己的職責)
        Orders order = new Orders();
        // ...建立訂單...
        ordersRepository.save(order);

        // 4. 透過 CouponService 處理優惠券使用狀態
        if (request.getCouponId() != null) {
            userCouponService.useCoupon(userId, request.getCouponId(), order.getOrderId());
        }

        // 5. 透過 CartService 清空購物車
        cartService.clearCart(userId);
        
        return order;
    }
}
```
這樣的改動不僅釐清了業務邏輯，也讓不同模組間可以獨立開發、測試與後續拆分。
