# 3. 多執行緒與併發架構分析報告

本報告旨在針對 Bookstore 專案在完成「模組化架構重構」以及「高併發風險修補」後，進行全面的多執行緒與併發存取架構分析。

## 一、 專案架構重構回顧

在本次重構中，我們將專案從傳統的「Controller / Service / Repository 單一分層架構」轉變為**「依功能模組劃分 (Feature-based Modular Architecture)」**。

*   **重構前問題**：所有模組的類別混雜在同一個 packages (`bookstore.controller`, `bookstore.service`, `bookstore.repository`)。導致模組邊界模糊，例如 `OrderService` 會直接依賴並修改 `BookRepository` 的狀態，形成高度耦合，在多執行緒環境下難以追蹤狀態變更的來源。
*   **重構後優勢**：資料夾結構已根據業務模組（如 `book`, `order`, `bookclub`, `user` 等）進行實體拆分。高內聚、低耦合的結構不僅限制了跨模組的不當資料存取，也為未來若需要進一步拆分為微服務（Microservices）打下良好基礎。

## 二、 核心併發風險修復與機制解析

### 1. 庫存管理：DB 原生行級鎖 (Row-Level Lock)

*   **業務場景**：使用者在購物車結帳 (`createOrderFromCart`) 或後台新增訂單 (`insertOrder`) 時，會觸發庫存扣除。
*   **重構前風險（Read-Modify-Write 競爭條件）**：
    原本程式碼在記憶體中進行庫存檢查與扣除：
    ```java
    // 舊版寫法 (危險)
    if (book.getStock() < cart.getQuantity()) { throw new BusinessException(...); }
    book.setStock(book.getStock() - cart.getQuantity());
    bookRepository.save(book);
    ```
    在高併發情況下，Thread A 和 Thread B 可能同時讀取到相同的庫存值（例如：庫存剩 1，兩人都想買 1 本）。兩者皆會通過檢查並將庫存設為 0，導致**超賣現像 (Overselling)**。
*   **重構後解法**：
    改用資料庫的**原生 UPDATE 行級鎖與條件約束**來確保原子性 (Atomicity)。
    ```java
    // BookRepository.java 新增方法
    @Modifying
    @Query("UPDATE BooksBean b SET b.stock = b.stock - :quantity WHERE b.bookId = :id AND b.stock >= :quantity")
    int deductStock(@Param("id") Integer id, @Param("quantity") Integer quantity);
    ```
    ```java
    // OrderService.java 調用
    int updatedRows = bookRepository.deductStock(book.getBookId(), cart.getQuantity());
    if (updatedRows == 0) { throw new BusinessException(...); }
    ```
*   **原理與多執行緒影響**：
    當多個請求同時嘗試購買同一本書時，關聯式資料庫 (MSSQL) 會在執行 `UPDATE` 語句時，針對該資料列自動加上排他鎖 (Exclusive Lock)。只有當第一個請求的交易 (Transaction) 提交或回滾後，後續請求才能獲得鎖。同時憑藉 `WHERE b.stock >= :quantity` 的條件，資料庫層級保證了庫存絕對不會被扣成負數。這種做法不阻礙系統對其他書籍的併發讀寫，效能極高且完全解決超賣問題。

### 2. 讀書會報名：悲觀鎖 (Pessimistic Locking)

*   **業務場景**：使用者報名讀書會 (`ClubRegistrationService.register`)，需檢查報名人數是否達到上限並增加參與人數。
*   **重構前風險**：
    報名人數的讀取與寫入分離，在高併發搶名額的情境下，如果同時有多人按下報名，可能會同時讀取到未滿額的狀態，造成報名人數超過 `maxParticipants` 的上限。
*   **重構後解法**：
    引入 Spring Data JPA 的悲觀寫入鎖 (`PESSIMISTIC_WRITE`)。
    ```java
    // BookClubsRepository.java 新增方法
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT b FROM BookClubsBean b WHERE b.clubId = :id")
    Optional<BookClubsBean> findByIdWithPessimisticLock(@Param("id") Integer id);
    ```
    ```java
    // ClubRegistrationService.java 調用
    @Transactional
    public ClubRegistrationsBean register(Integer clubId, Integer userId) {
        Optional<BookClubsBean> validClub = bookClubsRepository.findByIdWithPessimisticLock(clubId);
        // ... (檢查邏輯與人數上限判斷)
    }
    ```
*   **原理與多執行緒影響**：
    當 `@Transactional` 的 `register` 方法呼叫 `findByIdWithPessimisticLock` 時，JPA 會向資料庫發送帶有 `FOR UPDATE` (或 MSSQL 相當的鎖定語法如 `WITH (UPDLOCK, ROWLOCK)`) 的 SELECT 查詢。這會立即將該筆讀書會紀錄上鎖。
    其他執行緒若要在同一筆讀書會上進行報名，會在 `SELECT` 這一行被阻擋 (Block)，直到持有鎖的執行緒完成交易。這強力保證了從讀取人數、檢查是否滿額，到最終寫入報名表的整個過程是完全序列化的 (Serialized)，徹底阻斷了併發錯誤的發生。

## 三、 多執行緒與整體系統分析總結

經過上述調整後，系統對於多執行緒存取的防禦力得到了根本上的提升：

1.  **無狀態服務層設計 (Stateless Services)**：系統的 Service 層已經確認為無狀態（不包含可變的全域或實例變數），因此本身是 Thread-Safe 的。所有的併發狀態競爭都集中在外部資源（資料庫）。
2.  **鎖粒度控制得當 (Appropriate Lock Granularity)**：
    *   在處理**高頻率、高衝突**的「商品庫存」時，我們選擇了效能最高的 **DB Native Update (樂觀鎖的無鎖變體)**，將同步工作轉嫁給資料庫引擎做最高效的原子操作，最大化系統吞吐量。
    *   在處理**相對低頻但對一致性要求極高**的「活動報名 (讀書會)」時，採用了 **Pessimistic Lock (悲觀鎖)**。雖然這會導致針對同一場讀書會的報名請求排隊，但由於讀書會報名並非如電商秒殺那般頻繁的瓶頸，這種設計兼顧了實現難度與資料的絕對安全。
3.  **交易邊界 (Transaction Boundaries) 一致性**：所有修改庫存與建立訂單的方法，皆包裹在單一的 `@Transactional` 內，確保在發生任何異常（如餘額不足、其他物品缺貨）時，DB 原生鎖所扣除的庫存皆能正確 Rollback，符合 ACID 原則。

**結論**：重構後的專案不僅在模組劃分上更具備企業級系統的規範性，在多執行緒與併發存取的情境中，也能夠安全、穩定地處理高併發的訂單與報名請求，確保資料絕對的準確與安全。
